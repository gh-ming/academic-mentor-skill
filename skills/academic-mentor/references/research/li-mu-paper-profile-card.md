# Li Mu Paper Profile Card

Use this card when the mentor needs paper-level signals from Li Mu's technical teaching and execution-oriented writing style.

## Focus

This profile is most useful for:

- decomposition quality
- execution traction
- whether a paper leaves a runnable path
- whether the logic can be learned, implemented, and verified

## Introduction Pattern

Recurring pattern:

- introductions are strongest when they bridge concept and execution
- the reader should understand not only why the problem matters, but also why it is teachable and doable
- ambiguity is reduced early through explicit setup and stable terminology

What this implies for mentor behavior:

- ask whether the introduction leaves the reader with a clear route into the problem
- reject papers that sound impressive but remain too vague to implement
- reward writing that makes the problem understandable without dumbing it down

## Contribution Pattern

Recurring pattern:

- contributions should be decomposable into runnable steps
- the value of a method increases when the reader can connect concept, module, and experiment
- terminology should stay stable enough that another student could reproduce the route

What this implies for mentor behavior:

- ask whether each contribution can be turned into a concrete implementation or experiment
- cut abstract claims that leave no executable interpretation
- prefer papers whose contributions naturally imply the next milestone

## Experiment Pattern

Recurring pattern:

- experiments should include intermediate checks, not only final wins
- the reader should be able to see the first useful validation loop
- evidence is stronger when it helps both understanding and execution

What this implies for mentor behavior:

- ask what the first experiment proves before asking about the full system
- reward papers that make intermediate progress legible
- push for experiment ordering that mirrors actual implementation logic

## Limitation Pattern

Recurring pattern:

- if the plan is too big to execute, narrow it to the first reliable loop
- limitations should translate into next concrete steps, not vague future work

What this implies for mentor behavior:

- ask whether the current paper leaves a realistic next action
- convert overlarge plans into staged milestones

## Diagnostic Questions

- If a student had to implement this paper, what is the first correct step?
- Which experiment is the first executable proof rather than the final showcase?
- Are the contributions teachable and reproducible, or only verbally plausible?
- Does the limitation section imply the next milestone clearly enough?
